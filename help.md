## 2. SYSTEM ARCHITECTURE — WHY 8 AGENTS

### The Core Philosophy

Three architectural principles drive every design decision:

**Specialization over generalization:** Each agent has a single, well-defined responsibility. A specialist outperforms a generalist for each sub-task. Agent failures are contained and independently traceable.

**Validation before execution:** No recovery action reaches the approval queue without passing through the digital twin simulation. The system never proposes something it has not already validated.

**Auditability by design:** Every agent decision, reasoning trace, confidence score, and timestamp is logged. The runbook is a first-class output, not an afterthought.

### Full Pipeline

```
DATA SOURCES
├── Physics Simulator (digital twin engine)
├── OPSSAT-AD (real ESA telemetry — SENTINEL training)
└── JSON Event Logs (generated by simulator)
        │
        ▼
PHASE 1: WATCH (parallel)
├── SENTINEL ──── Isolation Forest + LSTM Autoencoder ──→ AnomalyEvent
├── VITALS ─────── Rolling trend + RUL estimation ────→ HealthReport
└── CHRONICLE ──── LLM event log correlation ──────────→ EventContext
        │
        ▼ (on AnomalyEvent trigger)
PHASE 2: DIAGNOSE
└── SHERLOCK ───── NetworkX causal graph + LLM CoT ──→ Diagnosis
        │
        ▼ (parallel execution)
┌───────────────────────────────────┐
│  ORACLE (Monte Carlo simulation)  │
│  ATHENA (recovery planning)       │
└───────────────────────────────────┘
        │
        ▼
PHASE 3: ACT & EXECUTE
├── GUARDIAN ──── Rule-based safety gate + step executor ──→ ApprovalDecision
└── QUARTERMASTER ── Resource & task scheduling ──────────→ ResourcePlan
        │
        ▼
PHASE 4: DOCUMENT
└── SCRIBE ─────── LLM + Jinja2 + python-docx ──────────→ PDF Runbook

OUTPUT: OPERATOR DASHBOARD (Streamlit UI)
```

### Why 8 Agents, Not Fewer?

The PS explicitly asks for anomaly detection, health assessment, resource scheduling, procedure execution, and runbook generation. Each is a distinct operational function requiring different tools, models, and reasoning approaches. Cramming them into fewer agents creates a monolithic system that is:
- Harder to debug (failure source unclear)
- Harder to audit (reasoning not isolated)
- Harder to explain to judges (no clear separation of concerns)
- Unsafe (safety-critical GUARDIAN logic mixed with LLM reasoning)

Separation of concerns is a **safety property**, not just a software pattern.

## 4. AGENT 1 — SENTINEL (Anomaly Detection)

**Purpose:** Continuously monitor telemetry streams, detect anomalies in real-time  
**Training data:** OPSSAT-AD (actually loaded and used)  
**Build time:** 4–6 hours

### Model A: Isolation Forest

**Why Isolation Forest?**
- O(n log n) complexity → real-time capable on streaming data
- Handles high-dimensional telemetry without feature engineering
- No labelled data required for training (unsupervised)
- scikit-learn implementation is production-proven

**Configuration:**
```python
from sklearn.ensemble import IsolationForest

model = IsolationForest(
    n_estimators=200,
    contamination=0.05,      # expect 5% anomalies in stream
    max_samples=256,
    random_state=42
)
# Train on 1,689 normal segments from dataset.csv
model.fit(normal_features)  # normal_features shape: (1689, 23)
```

**Expected performance on OPSSAT-AD:** Precision ~0.72, Recall ~0.68, F1 ~0.70

### Model B: LSTM Autoencoder

**Why LSTM-AE?**
- Captures temporal dependencies that IF misses
- Gradual voltage drift, periodic attitude oscillations — IF treats each window independently, LSTM remembers sequences
- Reconstruction error spikes on anomalous windows even when absolute values look normal (catches subtle drift)

**Architecture:**
```
Input: (batch, 128, 9)   ← 128 timesteps, 9 channels

Encoder:
  LSTM(128, hidden=128) → LSTM(64) → Dense(32)

Decoder (mirror):
  Dense(32) → LSTM(64) → LSTM(128) → TimeDistributed(Dense(9))

Output: (batch, 128, 9)  ← reconstruction

Loss: MSE between input and reconstruction
Anomaly score: reconstruction_error > (mean + 2.5 × std) on validation set
```

**Training:**
```python
# 30–50 epochs, Adam, lr=1e-3, trained on normal windows ONLY
# OPSSAT-AD: use segments where anomaly=0 only
# Sliding window: 128-sample window, step=10
```

**Training time:** ~45 min on GPU, ~3 hours on CPU (Fallback: 1D CNN autoencoder, trains in 15–20 min on CPU)

### Ensemble Logic

```python
def sentinel_detect(window_IF_score, window_LSTM_reconstruction):
    IF_flag = window_IF_score < threshold_IF          # negative = anomaly
    LSTM_flag = window_LSTM_reconstruction > threshold_LSTM
    
    if IF_flag and LSTM_flag:
        severity = calculate_severity(scores)
        return AnomalyEvent(severity=severity, confidence=high)
    elif IF_flag or LSTM_flag:
        # Log but don't alert — reduces false positives dramatically
        return None
    return None
```

**Key design choice:** Alert ONLY when BOTH models flag the same window. This dramatically reduces false positives — the #1 killer of operator trust in automated systems.

## 6. AGENT 3 — VITALS (Proactive Health Monitor)

**Purpose:** Continuous satellite health monitoring — proactive, not reactive  
**Key differentiator:** Runs even when no anomaly has fired. This transforms AERO-ASTRA from an alarm system into a mission ops layer.  
**Build time:** 3–4 hours

### How it works

**Per-parameter trend analysis:**
```python
# Rolling window slope detection
def detect_degradation(param_history: pd.Series, window_orbits=20):
    # 1 orbit ≈ 90 minutes for LEO
    window_samples = window_orbits * 90 * 6  # 6 samples/min
    recent = param_history.tail(window_samples)
    
    slope, intercept, r_value, p_value, std_err = scipy.stats.linregress(
        range(len(recent)), recent.values
    )
    
    # Negative slope in battery SOC max → degrading
    # Rising slope in panel temperature → thermal stress
    return slope, r_value**2  # slope + confidence (R²)
```

**Remaining Useful Life (RUL) estimation:**
```python
def estimate_rul(current_value, degradation_rate, threshold):
    # How many orbits before threshold breach?
    orbits_remaining = (current_value - threshold) / abs(degradation_rate)
    return orbits_remaining

# Example:
# battery_soc_max = 91.2%, degrading at 0.08%/orbit
# threshold = 85% (below this, heater performance degrades)
# RUL = (91.2 - 85) / 0.08 = 77.5 orbits ≈ 4.8 days
```

**Per-subsystem health score:**
```python
def subsystem_health_score(params: dict) -> float:
    # Weighted average of individual parameter health indicators
    # 0 = failed, 100 = nominal
    weights = {"battery_soc": 0.35, "panel_temp": 0.25, "attitude_error": 0.20, ...}
    scores = {p: calculate_parameter_health(v) for p, v in params.items()}
    return sum(weights[p] * scores[p] for p in weights)
```

### The "12+ Orbits" Story

When the demo shows "SAT-GAMMA panel_north degradation detected, threshold breach in ~12 orbits" with NO anomaly having fired — that is the most powerful moment in the entire presentation. It proves proactive intelligence that no threshold-based system can provide.

### Output Schema

```python
class HealthReport(BaseModel):
    satellite_id: str
    timestamp: datetime
    overall_score: float                        # 0-100
    subsystem_scores: Dict[str, float]          # per-subsystem
    degradation_alerts: List[DegradationAlert]  # trending issues
    rul_estimates: Dict[str, float]             # orbits remaining per param
```

## 7. AGENT 4 — SHERLOCK (Root Cause Diagnosis)

**Purpose:** Cross-subsystem causal inference — the PS explicitly calls out "cross-subsystem dependencies"  
**Key tech:** NetworkX causal dependency graph + LLM Chain-of-Thought reasoning  
**Build time:** 5–7 hours

### The Satellite Subsystem Dependency Graph

```python
import networkx as nx

G = nx.DiGraph()

# Nodes: satellite subsystems
subsystems = ["EPS", "TCS", "ADCS", "OBC", "COMMS", "PROP"]
G.add_nodes_from(subsystems)

# Edges: causal dependencies with weights (strength of coupling)
dependencies = [
    ("EPS", "TCS",   0.85),   # Power feeds thermal control
    ("EPS", "ADCS",  0.90),   # Power feeds attitude control
    ("EPS", "OBC",   0.95),   # Power feeds on-board computer
    ("EPS", "COMMS", 0.80),   # Power feeds communications
    ("TCS", "OBC",   0.70),   # Thermal affects compute performance
    ("ADCS", "COMMS",0.65),   # Attitude affects antenna pointing
    ("ADCS", "PROP", 0.75),   # Attitude errors trigger correction burns
    ("OBC", "EPS",   0.60),   # OBC commands power distribution
    ("OBC", "ADCS",  0.80),   # OBC controls attitude system
    ("OBC", "COMMS", 0.75),   # OBC controls communications
    ("OBC", "PROP",  0.70),   # OBC commands propulsion
]
for src, dst, weight in dependencies:
    G.add_edge(src, dst, weight=weight)
```

**Causal pattern (inspired by CATS benchmark structure):**
- Source nodes (EPS, PROP, ADCS) cause downstream degradation
- Output nodes (COMMS, TCS outputs) show symptoms
- The graph lets SHERLOCK ask: "if EPS is failing, which downstream systems are at risk and in what order?"

### Graph Traversal for Root Cause

```python
def trace_causal_chain(G, affected_subsystem: str) -> List[str]:
    # Find all ancestors (potential root causes) using BFS
    ancestors = nx.ancestors(G, affected_subsystem)
    
    # Rank by: betweenness centrality (most influential nodes)
    centrality = nx.betweenness_centrality(G, weight='weight')
    ranked_causes = sorted(ancestors, key=lambda n: centrality[n], reverse=True)
    
    return ranked_causes  # [EPS, OBC, ADCS, ...]

def find_downstream_impact(G, root_cause: str) -> List[str]:
    # Find all descendants (what will fail next)
    descendants = nx.descendants(G, root_cause)
    
    # Find shortest paths to estimate propagation time
    paths = {d: nx.shortest_path(G, root_cause, d, weight='weight') 
             for d in descendants}
    return paths
```

## 8. AGENT 5 — ORACLE (Digital Twin Simulator)

**Purpose:** Validate every recovery action BEFORE it is proposed to an operator  
**The PS says:** "simulates/validates outcomes using digital-twin context"  
**Key tech:** Custom Python SatelliteStateMachine + Monte Carlo simulation  
**Build time:** 4–6 hours

### Why This Is Your Biggest Technical Differentiator

Most teams skip ORACLE because it is harder to build. The teams that include it win technical depth points. A physics-based Monte Carlo simulation producing probability distributions is impossible to dismiss as a hallucination.

### Monte Carlo Process

```python
def oracle_simulate(satellite_state: SatelliteState, 
                    action: RecoveryAction, 
                    n_runs: int = 100) -> SimResult:
    
    outcomes = {
        "nominal": 0,
        "degraded": 0,
        "mission_loss": 0
    }
    final_states = []
    
    for run in range(n_runs):
        # Add Gaussian noise to initial state (real uncertainty)
        noisy_state = add_noise(satellite_state, sigma_pct=0.03)
        
        # Run simulation for 300 steps × 10 seconds = 50 minutes of sim time
        sim_state = noisy_state.copy()
        for step in range(300):
            sim_state = apply_action_step(sim_state, action, step)
            sim_state = apply_physics(sim_state)              # orbital dynamics
            sim_state = add_step_noise(sim_state, sigma=0.02) # sensor noise
            
            # Check constraint violations at each step
            if check_mission_loss(sim_state):
                outcomes["mission_loss"] += 1
                break
        else:
            # Simulation completed without loss
            if check_nominal_recovery(sim_state):
                outcomes["nominal"] += 1
            else:
                outcomes["degraded"] += 1
        
        final_states.append(sim_state)
    
    return SimResult(
        nominal_recovery_prob=outcomes["nominal"] / n_runs,
        degraded_prob=outcomes["degraded"] / n_runs,
        mission_loss_prob=outcomes["mission_loss"] / n_runs,
        time_to_nominal=estimate_recovery_time(final_states),
        constraint_violations=check_all_constraints(final_states),
        side_effects=identify_side_effects(satellite_state, final_states)
    )
```

**For critical actions (irreversible commands):** n_runs = 300, sigma_pct = 0.05

### Output Schema

## 9. AGENT 6 — ATHENA (Recovery Planner)

**Purpose:** Generate ranked recovery options using LLM Chain-of-Thought reasoning  
**Build time:** 4–6 hours

### How it works

```python
ATHENA_SYSTEM_PROMPT = """

Given a diagnosis, simulation results, and mission constraints,
generate exactly 3 recovery options ranked by safety × effectiveness.

Mission constraints to respect:

- Communication windows: {next_contact_UTC}
- Orbit constraints: {maneuver_window}

Output ONLY valid JSON:
{
  "options": [
    {
      "name": string,
      "procedure_steps": [string],
      "safety_score": float (0-1),
      "effectiveness_score": float (0-1),
      "resource_cost": float (normalized 0-1),
      "predicted_outcome": string,
      "contra_indications": [string]
    }
  ],

}
"""
```

### Anti-Hallucination Measures

1. **Never send raw telemetry to ATHENA** — only structured facts (anomaly type, delta values, diagnosis)
2. **Pydantic validation** — every ATHENA output validated against schema; malformed responses trigger automatic retry with corrected prompt
3. **ORACLE post-validation** — every option ATHENA generates is run through ORACLE before reaching GUARDIAN; physically impossible options are rejected
4. **Temperature setting** — use `temperature=0.1` for all production ATHENA calls to minimize randomness

### Output Schema

```python

IRREVERSIBLE_COMMANDS = [
    "EPS_MAIN_BUS_SWITCH", "SAFE_MODE_ENTRY", "EMERGENCY_SHUTDOWN",

def guardian_evaluate(plan: RecoveryPlan) -> ApprovalDecision:
    option = plan.options[plan.recommended_option_id]
    
    # Rule 1: Safety score too low
    if option.safety_score < 0.85:
        return ApprovalDecision(route="HUMAN", reason="Safety score below threshold")
    
    # Rule 2: Irreversible command
    for step in option.procedure_steps:
        if any(cmd in step for cmd in IRREVERSIBLE_COMMANDS):
            return ApprovalDecision(route="HUMAN", reason="Irreversible command requires human sign-off")
    
    # Rule 3: Novel scenario (SHERLOCK uncertain)
    if diagnosis.confidence < 0.70:
        return ApprovalDecision(route="HUMAN", reason="Diagnosis confidence too low")
    
    # Rule 4: ORACLE simulation shows risk
    if option.oracle_nominal_prob < 0.80:
        return ApprovalDecision(route="HUMAN", reason="Simulation outcome uncertain")
    
    # Auto-approve ONLY: all rules pass
    if (option.safety_score > 0.95 and 
        option.oracle_nominal_prob > 0.90 and 
        diagnosis.confidence > 0.90):
        return ApprovalDecision(route="AUTO", reason="All safety thresholds exceeded")
    
    return ApprovalDecision(route="HUMAN", reason="Precautionary human review")
```

### Z3 Formal Verification

```python
from z3 import Solver, Real, And, Or, sat

def verify_constraints_z3(action: RecoveryAction, state: SatelliteState) -> bool:
    solver = Solver()
    
    # State variables
    battery_soc = Real('battery_soc')
    panel_temp = Real('panel_temp')
    fuel = Real('fuel')
    
    # Current state constraints
    solver.add(battery_soc == state.battery_soc)
    solver.add(panel_temp == state.panel_temp_north)
    solver.add(fuel == state.fuel_remaining_kg)
    
    # Mission safety constraints (formal specification)
    solver.add(battery_soc >= 20.0)       # Never discharge below 20%
    solver.add(panel_temp <= 65.0)        # Max panel temperature
    solver.add(fuel >= 0.5)              # Always keep 0.5kg reserve
    
    # Action effect constraints
    solver.add(battery_soc >= battery_soc - action.power_draw * action.duration)
    
    # Check satisfiability — if UNSAT, action violates constraints
    return solver.check() == sat
```

### Key Physics Modelled

```python
def step(self, dt=10):  # 10-second simulation step
    """Single physics step: update all subsystem states."""
    
    self.mission_time += dt
    orbit_phase = (self.mission_time % 5400) / 5400  # 90-min LEO orbit
    
    # 1. Orbital thermal cycling
    if orbit_phase < 0.35:  # eclipse (35% of orbit)
        self.eclipse_state = True
        self.solar_array_current = 0.0
        # Panels cool during eclipse
        self.panel_temp_north += (-40 - self.panel_temp_north) * 0.001 * dt
    else:
        self.eclipse_state = False
        self.solar_array_current = 4.2  # A in full sun
        # Panels heat in sunlight
        self.panel_temp_north += (55 - self.panel_temp_north) * 0.0008 * dt
    
    # 2. Battery dynamics (Li-ion charge/discharge)
    if self.eclipse_state:
        # Discharge during eclipse
        discharge_rate = self.load_current / (capacity_Ah * 3600) * 100
        self.battery_soc -= discharge_rate * dt
        # Li-ion voltage curve (nonlinear below 20% SOC)
        self.battery_voltage = liion_voltage_curve(self.battery_soc)
    else:
        # Charge in sunlight
        charge_rate = (self.solar_array_current - self.load_current) / (capacity_Ah * 3600) * 100
        self.battery_soc = min(100, self.battery_soc + charge_rate * dt)
    
    # 3. Thermal coupling (EPS heat → TCS)
    heat_dissipation = self.load_current * self.bus_voltage * 0.05  # 5% resistive heating
    self.battery_temp += heat_dissipation * 0.001 * dt
    
    # 4. Heater logic
    for i, heater in enumerate(self.heater_state):
        if self.battery_temp < 5.0 and not heater:
            self.heater_state[i] = 1  # Activate heater
            self.load_current += 0.4  # Heater draws 0.4A
        elif self.battery_temp > 35.0 and heater:
            self.heater_state[i] = 0  # Deactivate
            self.load_current -= 0.4
    
    # 5. Reaction wheel momentum accumulation (ADCS)
    external_torque = 1.2e-7  # N·m (gravity gradient, SRP)
    for i in range(3):
        self.rw_rpm[i] += (external_torque / rw_moment_of_inertia) * dt * 60 / (2*pi)
        # Desaturation burn needed when RPM > 4500
```

### Scenario 1: EPS Power Spike
- **Trigger:** Solar array partial short circuit
- **Initial telemetry:** battery_current: +28% from baseline, battery_soc: dropping at 0.8%/min
- **SENTINEL response:** IF flags battery_current outlier, LSTM-AE flags current spike temporal pattern → ALERT at t+14s
- **CHRONICLE:** "2 load shedding events detected in prior 20 minutes" — corroborated
- **SHERLOCK:** EPS → TCS → ADCS causal chain: battery over-discharge → thermal buildup (heaters overcompensate) → ADCS power budget starvation in ~40 min
- **ORACLE:** Option 2 (targeted power rerouting) shows 87% nominal recovery, 48-min return to full ops

## 20. JUDGE Q&A — TECHNICAL ANSWERS

**Q: Is this trained on real satellite data?**
Yes, for SENTINEL. OPSSAT-AD — real CubeSat telemetry from a satellite that flew in space, annotated by ESA ESOC spacecraft operations engineers, published Nature Scientific Data 2025. F1 ~0.78 on 78,315 held-out test rows, compared against 30 published baselines.

**Q: Why synthetic data for the demo?**
Every public satellite telemetry dataset anonymizes channel names — OPSSAT-AD channels are CADC0872, CADC0892 etc. An operator cannot respond to "CADC0872 exceeded threshold." They need "Battery voltage dropped to 22.3V." Our physics simulator provides human-readable subsystem data following DLR/GSOC operational methodology.

**Q: Did you train on CATS?**
No. CATS is an industrial control system simulation, not a satellite. Its causal structure — source sensors causing downstream degradation — informed how we designed fault injection in our satellite simulator. SHERLOCK validation uses simulator data where we know ground truth because we injected the fault.

**Q: What if the LLM hallucinates?**
Three independent safeguards: ORACLE validates every action through physics simulation (physically impossible proposals fail), GUARDIAN enforces rule-based safety gates (not LLM) for critical decisions, Z3 formally verifies no AI command violates mission constraints. LLMs suggest; physics validates; humans approve.

**Q: How does this scale to 50+ satellites?**
SENTINEL runs one stateless instance per satellite — horizontally scalable. SHERLOCK and ORACLE are stateless services. QUARTERMASTER handles cross-satellite redistribution natively. We demonstrate 3; the design requires no architectural changes for 50+.

**Q: Why 8 agents, not one big LLM?**
Specialization: each agent has tight scope and specific tools. Parallelism: SENTINEL + CHRONICLE + VITALS run concurrently. Auditability: each agent's reasoning is logged separately. Safety: GUARDIAN is deliberately NOT an LLM. Separation of concerns is a safety property.

**Q: Why is GUARDIAN not an LLM?**
Safety-critical operations require deterministic, auditable decision logic. An LLM might reason "the situation seems low risk" and auto-approve a dangerous command. A rule engine either satisfies the constraint or it doesn't — no ambiguity, no reasoning errors, no hallucination. This mirrors how NASA and ESA actual approval workflows are designed.
